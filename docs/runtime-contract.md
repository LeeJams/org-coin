# Runtime Contract

Live trading is available only behind an explicit BTC-only runtime gate. Exchange secrets must still come from environment variables or a secret store only.

`src/runtime/config.ts` is the code entrypoint for this contract. It merges `.env` and `process.env`, keeps paper-safe defaults when optional safety values are unset, and fails fast if live execution is enabled.

## Required mode flags

- `TRADING_MODE`: `dry_run`, `paper`, or `live`
- `ENABLE_LIVE_EXECUTION`: must be `true` only when `TRADING_MODE=live`

Current enforced behavior:

- `TRADING_MODE=live` requires `ENABLE_LIVE_EXECUTION=true`
- `ENABLE_LIVE_EXECUTION=true` requires `TRADING_MODE=live`
- `TRADING_MODE=live` requires `BITHUMB_ACCESS_KEY` and `BITHUMB_SECRET_KEY`
- Managed live service is restricted to `DRY_RUN_MARKETS=KRW-BTC`
- Managed live service requires `LIVE_READINESS_APPROVED=true` plus
  `LIVE_READINESS_SUMMARY_PATH` pointing to a single-candidate readiness
  artifact whose `strategyAssessment.classification` is `live_candidate`
- `LIVE_READINESS_SUMMARY_PATH` must include a valid `generatedAt` timestamp
  and must be fresh under `LIVE_READINESS_MAX_AGE_MS` (default 15 minutes).
  PM2 live start scripts refresh the readiness artifact before startup.
- Managed live service also requires `LIVE_GOAL_STATUS_PATH` to point to a
  fresh goal-status artifact generated after the readiness summary it
  authorizes. Its completion audit must be achieved with no failed criteria, no
  listed missing requirements, an explicit zero missing-requirement count, and
  a `criteria` list whose failed IDs match `failedCompletionCriteria` before
  live config loading can proceed. BTC PM2 live start/restart scripts therefore
  refresh the BTC readiness summary before rerunning the live-goal gate.
- Spot-perp carry live execution also requires `--live-goal-status` to point to
  the current live-goal artifact. That artifact must be fresh and its
  `generatedAt` must be at least as recent as the carry/readiness reports being
  used, so a stale approval cannot authorize newer spot-perp evidence. Its
  completion audit must be achieved with no failed criteria, no listed missing
  requirements, an explicit zero missing-requirement count, and a consistent
  `criteria` list.
- Cross-exchange relative-value live execution uses the same goal-status rule:
  the live-goal artifact must be fresh, must be generated after the readiness
  and operational-proof evidence it authorizes, and must carry an achieved
  completion audit with no remaining failed or missing requirements and a
  consistent `criteria` list.
- Managed live service also requires
  `LIVE_TRADING_FEE_SCHEDULE_CONFIRMED=true` and
  `LIVE_TRADING_FEE_ROUND_TRIP_BPS` set to the active account's KRW-BTC
  round-trip fee. For BTC 240-minute readiness, this value must be no higher
  than the benchmark's `feeRoundTripBps`, and live startup checks the private
  Bithumb order-chance fee before syncing account state.
- PM2 live targets default those approval and fee-confirmation flags to `false`,
  but the ecosystem file reads explicit environment overrides. Live startup
  therefore requires an intentional operator command with those variables set
  after the readiness gates pass.
- `LIVE_READINESS_SUMMARY_PATH` may point to either a single-candidate
  `summarize-dry-run-returns` output or the BTC 240-minute time-series
  readiness artifact generated by `refresh-btc-240m-momentum-readiness`.
  Multi-candidate audit artifacts such as `audit-live-candidates` output are
  review evidence only and are rejected by the managed live startup gate.
- BTC 240-minute time-series readiness is live-eligible only when all live
  checks pass, including a realized reduce-only paper exit, no remaining paper
  position, `realizedExitReusePolicy=true`, and
  `positiveRealizedPaperExitPnl=true`. The exit proof must be the first
  reduce-only exit recorded for the entry signal. A clean but losing paper exit
  remains paper-only.
- `DRY_RUN_ENTRY_PROFILE=btc_240m_momentum_min75_candidate_v1` is supported
  only for BTC 240-minute readiness evidence whose candidate parameters include
  `minReturnBps=75`; it is a replacement paper/live path candidate, not the
  default live profile.
- Managed live startup also requires `LIVE_GOAL_STATUS_PATH` to select the
  same strategy family as `DRY_RUN_ENTRY_PROFILE`: public BTC 240m uses
  `selectedLiveCandidate.type=btc_240m_momentum`, and the min75 profile uses
  `selectedLiveCandidate.type=btc_240m_momentum_min75`. Paper-only candidate
  labels such as `btc_240m_momentum_min75_paper_candidate` cannot authorize the
  live process.
- PM2 live start/restart scripts must run
  `dry-run:audit-live-goal-process-alignment` after the strategy-specific
  review or goal gate and before `npm run build && pm2 ...`, so a completed
  live-goal audit cannot be bypassed by stale or unapproved live processes.
- Process alignment treats a stale live-goal completion audit as a violation
  even when live startup is blocked. The live-goal status artifact must include
  `completionAudit.failedCompletionCriteria`, `completionAudit.missingRequirements`,
  and an integer `completionAudit.missingRequirementCount` that matches the
  missing-requirements array length before PM2 alignment can be considered
  current. `failedCompletionCriteria` contains stable criterion IDs from the
  completion-audit `criteria` list and must match the criteria whose
  `passed` value is not true. The `criteria` list must also include the live
  goal's required core IDs (`candidate_selected_from_current_evidence`,
  `profitability_evidence_satisfied`, `known_losing_paths_rejected`,
  `current_entry_sanity_clear`, `no_current_focus_recompare_caution`,
  `live_startup_gate_allowed`), while `criteria[].criterion` remains the
  human-readable explanation. `candidate_selected_from_current_evidence` is
  about the live-authorizing candidate, not a research-only focus; it fails
  while `selectedLiveCandidate` is absent. `live_startup_gate_allowed` likewise
  fails until `liveStartupAllowed=true`, no blockers remain, and the selected
  live candidate is populated.
- A spot-perp carry status with
  `processControlPlan.carryResearch.desiredState=recompare_challenger_before_live_review`
  is still a blocked live-goal state. Any
  `liveStartupPlan.recompareChallengerPlan` commands are research-focus
  handoff commands only; live startup remains unauthorized until the challenger
  has its own passing readiness evidence and the global live-goal audit passes.
- `npm run dry-run:review-btc-240m-min75-live-ready` is the review-only
  command for the min75 path after its scheduled paper exit. It runs the min75
  strategy gate and the top-level live-goal gate without starting a PM2 live
  process. Use `pm2:start:live-btc-min75` only after this review path succeeds
  and the operator intentionally proceeds with live startup.
- `npm run dry-run:review-pieverse-60m-reversal-lb168-live-ready` is the
  review-only command for the current `KRW-PIEVERSE` replacement research path.
  It pauses the PIEVERSE dry-run observer to avoid concurrent writes, runs the
  sequential refresh CLI directly, runs the replacement readiness gate, reruns
  the top-level live-goal gate, and restarts only the PIEVERSE dry-run observer.
  It does not mark the live execution path ready; PIEVERSE remains blocked
  until a live runner is separately implemented and tested.
- Spot-perp carry live-readiness review scripts use the canonical 72-hour carry
  report for market and inventory scope, and pass the matching
  `*-fee-stress-25bps-latest.json` report as operational proof fee budget. This
  keeps private account fee checks aligned with the fee-stressed carry evidence
  and records the applied budget under
  `operationalReadiness.operationalProof.feeBudget`; it does not bypass
  credentials, margin, inventory, observation-span, spread-control, or top-level
  live-goal gates.
- `TRADING_MODE=dry_run` uses optimistic internal fills at the reference price with zero fee or liquidity impact
- `TRADING_MODE=paper` uses the slippage- and fee-aware simulator for more conservative replay estimates
- `TRADING_MODE=live` routes orders through Bithumb private REST with explicit account and order-chance preflight

## Operational safety inputs

- `MAX_DAILY_LOSS_KRW`
- `MAX_ORDER_NOTIONAL_KRW`
- `MAX_POSITION_NOTIONAL_KRW`
- `MAX_POSITION_NOTIONAL_BY_MARKET_JSON`
- `DATA_STALE_AFTER_MS`
- `KILL_SWITCH_REJECT_STREAK`
- `PAPER_SESSION_ARTIFACTS_DIR`

Current runner semantics:

- `MAX_DAILY_LOSS_KRW`, `MAX_ORDER_NOTIONAL_KRW`, `DATA_STALE_AFTER_MS`, and `KILL_SWITCH_REJECT_STREAK` fall back to built-in paper defaults when unset
- `MAX_POSITION_NOTIONAL_KRW=0` keeps the built-in per-market paper caps; any positive value overrides all market caps uniformly
- `MAX_POSITION_NOTIONAL_BY_MARKET_JSON` merges market-specific cap overrides on top of the default or global cap baseline
- `PAPER_SESSION_ARTIFACTS_DIR` controls where persisted session reports, reject ledgers, and NDJSON ledger evidence are written
- `TRADING_MODE=live` narrows the approved market universe to `KRW-BTC`

## Managed dry_run service inputs

These values drive the `pm2`-managed `dry_run` loop in `dist/src/cli/run-dry-run-service.js`.

- `DRY_RUN_BASE_DIR`
- `DRY_RUN_ENTRY_PROFILE`
- `DRY_RUN_EXIT_PROFILE`
- `DRY_RUN_SYNTHETIC_EXIT_POLICY`
- `DRY_RUN_INITIAL_CASH_KRW`
- `DRY_RUN_LOOP_INTERVAL_SECONDS`
- `DRY_RUN_LOG_DIR`
- `DRY_RUN_CYCLE_LOG_FILE`
- `DRY_RUN_PYTHON_BIN`
- `DRY_RUN_MARKETS`
- `DRY_RUN_FRESHNESS_SLA_MS`
- `DRY_RUN_CANDLE_COUNT`
- `DRY_RUN_TRADE_COUNT`
- `DRY_RUN_WS_SECONDS`
- `DRY_RUN_TRADE_WARMUP_SECONDS`
- `DRY_RUN_BOOTSTRAP_ITERATIONS`
- `DRY_RUN_BOOTSTRAP_INTERVAL_SECONDS`
- `DRY_RUN_WS_CHANNELS`
- `LIVE_READINESS_APPROVED` for live mode only
- `LIVE_READINESS_SUMMARY_PATH` for live mode only
- `LIVE_READINESS_MAX_AGE_MS` for live mode only
- `LIVE_TRADING_FEE_SCHEDULE_CONFIRMED` for live mode only
- `LIVE_TRADING_FEE_ROUND_TRIP_BPS` for live mode only

Current managed-service semantics:

- The service captures public market data, builds a replay scenario, executes it in `dry_run`, and appends one structured summary line per cycle to `DRY_RUN_LOG_DIR/DRY_RUN_CYCLE_LOG_FILE`
- The PM2 ecosystem file also keeps stdout/stderr under `var/log/pm2/`
- `TRADING_MODE=paper` can remain the repo default in `.env`; the managed service forces `dry_run` for its own execution path
- Exchange secrets are ignored outside live mode
- `DRY_RUN_EXIT_PROFILE` switches among dry-run-only exit evaluators without changing the entry gate profile
- `balanced_v1_book_confirm3_trend_hold` is a paper-only hold candidate that
  defers the 15-minute time stop while the current 5-minute BTC return remains
  positive; it is not a live bypass
- `balanced_v1_book_confirm3_trend_hold_guarded` is the same paper-only hold
  experiment with the positive-trend time-stop deferral capped at 30 minutes,
  so paired replay can detect open-risk migration instead of treating deferred
  exits as live readiness
- `DRY_RUN_SYNTHETIC_EXIT_POLICY` controls how a session with an unresolved position is marked at the end of the replay
- Return summaries from `src/cli/summarize-dry-run-returns.ts` include BTC
  buy-and-hold benchmark, excess return, drawdown, and live-readiness gates.
  Operationally stable paper evidence is not enough for live mode unless this
  benchmark classifies the strategy as `live_candidate`.
- BTC 240-minute momentum readiness from
  `src/cli/refresh-btc-240m-momentum-readiness.ts` is a separate BTC-only
  single-candidate path. It blocks live until the open paper position has a
  realized reduce-only exit, that exit reconciles to no open position, and the
  realized paper exit PnL is positive.
- The stricter `btc_240m_momentum_min75_candidate_v1` entry profile uses the
  same 240-minute BTC managed path but requires matching readiness evidence
  with `minReturnBps=75`.
- For min75, run `npm run dry-run:review-btc-240m-min75-live-ready` before any
  live PM2 command. It refreshes min75 readiness and the top-level goal audit
  without starting `live-btc-min75-manager`. The min75 live start/restart
  scripts also run this review command before PM2 startup, so a stale top-level
  goal report cannot block a freshly realized min75 exit before it is rechecked.
- Aggregate audits from `src/cli/audit-live-candidates.ts` can document which
  candidates are blocked, but they cannot be used as `LIVE_READINESS_SUMMARY_PATH`.

For the current paper-only repo state, leave `BITHUMB_ACCESS_KEY` and `BITHUMB_SECRET_KEY` blank in the local `.env` file.

None of these secrets or runtime values should be stored in source, Paperclip issue bodies, or plan documents.
